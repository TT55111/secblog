# 增删功能工作流

本规则定义了在已有网站项目上增删功能的标准工作流，所有功能增删必须按以下 6 个阶段顺序执行。核心原则：**回归测试优先，系统化调试，禁止凭直觉改代码**。

---

## 阶段 01：🧠 需求分析（Requirement Analysis）

**触发条件**：需要在已有项目上增加或删除功能时。

**执行要求**：
- 使用 `brainstorming` skill 进行需求分析，但比新项目更轻量
- 必须评估以下内容：
  - **变更范围**：增删什么功能？涉及哪些模块？
  - **影响分析**：对现有功能的影响范围，哪些模块可能被波及？
  - **依赖梳理**：新功能与已有代码的依赖关系
  - **删除风险**：删除功能时，是否有其他模块依赖它？
- 产出变更说明文档，保存至 `docs/feature-changes/YYYY-MM-DD-<feature>-change.md`
- 变更说明文档必须包含：变更描述、影响范围、受影响的已有模块清单、风险评估

**产出物**：`docs/feature-changes/YYYY-MM-DD-<feature>-change.md`

---

## 阶段 02：🌿 隔离工作区（Git Worktrees）

**触发条件**：需求分析完成后，开始编码前。

**执行要求**：
- 使用 `using-git-worktrees` skill 创建隔离工作区
- 从主分支创建新的特性分支
- 工作区目录：`.worktrees/<feature>/`
- 确保工作区干净，建立测试基线
- **回归基线**：记录当前所有已有测试的通过状态，作为回归对比的基准
- 验证工作区环境可正常运行（构建、测试通过）

**产出物**：`.worktrees/<feature>/` 隔离工作区 + 回归测试基线

---

## 阶段 03：📋 编写计划（Writing Plans）

**触发条件**：工作区就绪后，开始实施前。

**执行要求**：
- 使用 `writing-plans` skill 将变更拆分为可执行的小任务
- 每个任务必须明确：目标、输入、输出、验收标准
- **额外要求**：每个任务必须标注**可能影响的已有模块**，明确回归验证范围
- 任务粒度：单个任务应在合理时间内可完成
- 标注任务间依赖关系，识别可并行的任务
- 保存至 `docs/feature-changes/YYYY-MM-DD-<feature>-plan.md`

**产出物**：`docs/feature-changes/YYYY-MM-DD-<feature>-plan.md`

---

## 阶段 04：🛠️ 实施与调试（Implementation & Debugging）

**触发条件**：计划编写完成并获得确认后。

**执行要求**：

### TDD 流程
- 使用 `test-driven-development` skill 强制执行 TDD 流程
- **回归测试优先**：先编写回归测试确保旧功能不受影响，再编写新功能测试
- 严格遵循 RED → GREEN → REFACTOR 循环：
  1. **RED**：先编写失败的测试用例（回归测试 + 新功能测试）
  2. **GREEN**：编写最少代码使测试通过
  3. **REFACTOR**：重构代码，消除重复，提升质量，确保测试仍通过
- 每个循环完成后才能进入下一个循环
- 禁止在 RED 阶段跳过测试直接写实现

### 调试规则
- **遇到 bug 时，禁止凭直觉修改代码**
- 必须使用 `systematic-debugging` skill 进行系统化排查：
  1. **复现问题**：确认 bug 可稳定复现
  2. **缩小范围**：通过日志、断点等手段缩小问题范围
  3. **定位根因**：找到 bug 的根本原因，而非表面症状
  4. **验证修复**：修复后确认 bug 已解决且未引入新问题
- 每一步都必须有证据支撑，不能靠猜测

### 浏览器验证
- 单元测试通过后，必须使用 `webapp-testing` skill 在浏览器中实际验证页面功能
- 验证范围包括：新功能 + 受影响的旧功能

**产出物**：通过所有测试（回归 + 新功能 + 浏览器验证）的代码

---

## 阶段 05：✅ 验证与审查（Verification & Review）

**触发条件**：实施与调试完成后，合并前。

**执行要求**：
- **浏览器全量回归验证**：使用 `webapp-testing` skill 对所有关键页面和功能进行浏览器级别的回归验证，确保旧功能未受影响
- **安全审查**：使用 `security-best-practices` skill 对变更代码进行安全维度审查，新增的接口和功能可能引入新的安全风险
- **代码审查**：使用 `requesting-code-review` skill 发起代码审查
  - 审查维度：代码质量、架构合理性、安全性、性能、可维护性
  - 额外关注：变更是否超出预期范围、是否有不必要的副作用
- 严重级别分类：
  - **CRITICAL**：必须修复，阻塞合并（安全漏洞、数据丢失风险、回归破坏）
  - **IMPORTANT**：强烈建议修复（性能问题、潜在 bug、设计缺陷）
  - **MINOR**：可选修复（代码风格、命名、注释改进）
- CRITICAL 级别问题必须修复后才能继续

**产出物**：通过全量回归验证 + 安全审查 + 代码审查的代码

---

## 阶段 06：🏁 收尾分支（Finish Branch）

**触发条件**：验证与审查全部通过后。

**执行要求**：
- **完成前验证**：使用 `verification-before-completion` skill，在声称工作完成之前必须运行验证命令（构建、测试、lint 等）并确认输出通过，禁止未经实际验证就声称完成
- 使用 `finishing-a-development-branch` skill 处理分支生命周期
- 可选操作：
  - **merge**：合并到主分支（适用于已验证的完整功能）
  - **PR**：创建 Pull Request（适用于需要团队审查的场景）
  - **keep**：保留分支继续开发（适用于功能未完全完成）
  - **discard**：丢弃分支（适用于方案废弃或实验失败）
- 合并前确保：所有测试通过（含回归测试）、无 CRITICAL/IMPORTANT 级别审查问题、文档已更新
- 清理工作区：删除 `.worktrees/<feature>/` 目录

---

## 工作流执行规则

1. **顺序执行**：阶段 01-06 必须按顺序执行，不可跳过任何阶段
2. **阶段门控**：每个阶段必须有明确的产出物，才能进入下一阶段
3. **回退机制**：若实施阶段发现需求理解有误，应回退到阶段 01 重新分析
4. **回归优先**：所有测试环节必须先验证旧功能不受影响，再验证新功能正确性
5. **禁止直觉调试**：遇到 bug 必须使用 `systematic-debugging` 系统化排查，禁止凭直觉修改代码
6. **小变更豁免**：对于非常简单的增删（如修改文案、调整配置），可跳过阶段 01-03，直接从阶段 04 开始
7. **紧急修复豁免**：紧急 hotfix 可跳过阶段 01-03，但必须执行阶段 04-06
