# 开发网站工作流

本规则定义了项目开发的标准工作流，所有功能开发、重构和重大变更必须按以下 8 个阶段顺序执行。

---

## 阶段 01：🧠 设计构思（Brainstorming）

**触发条件**：接到新功能或重大变更需求时，必须先进入此阶段。

**执行要求**：
- 使用 `brainstorming` skill 进行需求分析和设计构思
- 理清需求背景、目标用户、核心功能、边界条件
- **视觉伴侣（Visual Companion）**：
  - 当预判后续问题会涉及视觉内容（模型图、布局、架构图等）时，必须向用户单独提供视觉伴侣选项
  - 提供视觉伴侣的消息必须**独立发送**，不得与澄清问题或其他内容合并
  - 提示语示例："我们接下来要讨论的一些内容，如果能在浏览器中展示可能会更容易理解。我可以制作模型图、图表、对比图等视觉内容。要试试吗？（需要打开一个本地 URL）"
  - 用户同意后，针对每个问题**独立判断**使用浏览器还是终端：
    - **浏览器**：模型图、线框图、布局对比、架构图、视觉设计对比
    - **终端**：需求问题、概念选择、权衡列表、范围决策
  - 判断标准：用户看到视觉内容是否比阅读文字更容易理解？
  - 用户同意后，需阅读详细指南 `skills/brainstorming/visual-companion.md`
- 产出设计文档，保存至 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
- 设计文档必须包含：需求描述、方案选项、技术选型、风险点

**产出物**：`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`

---

## 阶段 02：🌿 隔离工作区（Git Worktrees）

**触发条件**：设计构思完成后，开始编码前。

**执行要求**：
- 使用 `using-git-worktrees` skill 创建隔离工作区
- 从主分支创建新的特性分支
- 工作区目录：`.worktrees/<feature>/`
- 确保工作区干净，建立测试基线
- 验证工作区环境可正常运行（构建、测试通过）

**产出物**：`.worktrees/<feature>/` 隔离工作区

---

## 阶段 03：📋 编写计划（Writing Plans）

**触发条件**：工作区就绪后，开始实施前。

**执行要求**：
- 使用 `writing-plans` skill 将设计拆分为可执行的小任务
- 每个任务必须明确：目标、输入、输出、验收标准
- 任务粒度：单个任务应在合理时间内可完成
- 标注任务间依赖关系，识别可并行的任务
- 保存至 `docs/superpowers/plans/YYYY-MM-DD-<feature>.md`

**产出物**：`docs/superpowers/plans/YYYY-MM-DD-<feature>.md`

---

## 阶段 04：🤖 分发任务（Subagent-Driven）

**触发条件**：计划编写完成并获得确认后。

**执行要求**：
- 使用 `subagent-driven-development` skill 分发任务
- 每个任务分配一个独立 subagent 执行
- 无依赖关系的任务可并行执行
- 每个任务执行两阶段审查：
  - **第一阶段**：任务完成后的自检（功能正确性）
  - **第二阶段**：跨任务的集成审查（一致性、兼容性）

---

## 阶段 05：🎨 前端设计与开发（Frontend Design）

**触发条件**：当项目涉及用户界面、页面布局、交互设计等前端内容时，必须进入此阶段。纯后端/无 UI 项目可跳过。

**执行要求**：
- 根据项目类型选择前端技能：
  - **`frontend-design`**：适用于需要视觉冲击力和独特性的场景（品牌展示、创意页面、产品发布页、艺术化 UI）
  - **`frontend-skill`**：适用于需要专业克制和功能优先的场景（SaaS 落地页、应用界面、仪表盘、管理后台、工具类 UI）
  - 两者可组合使用：`frontend-design` 确定整体美学方向，`frontend-skill` 约束具体实现细节
- **视觉伴侣（Visual Companion）**：
  - 前端设计阶段**默认启用视觉伴侣**，无需再次询问用户
  - 所有视觉方案（布局、配色、组件样式、交互效果）必须通过浏览器展示给用户确认
  - 每次视觉方案变更或迭代，都必须在浏览器中实时呈现
  - 用户的视觉反馈优先于文字描述——"看起来对不对"比"逻辑对不对"更重要
- 产出视觉原型，获得用户对外观和交互的确认后，再进入 TDD 阶段补充逻辑和测试
- 视觉确认前禁止进入 TDD 阶段

**产出物**：经用户确认的前端视觉原型

---

## 阶段 06：🧪 TDD 实施（Test-Driven Development）

**触发条件**：每个 subagent 执行编码任务时。

**执行要求**：
- 使用 `test-driven-development` skill 强制执行 TDD 流程
- 严格遵循 RED → GREEN → REFACTOR 循环：
  1. **RED**：先编写失败的测试用例，明确定义期望行为
  2. **GREEN**：编写最少代码使测试通过
  3. **REFACTOR**：重构代码，消除重复，提升质量，确保测试仍通过
- 每个循环完成后才能进入下一个循环
- 禁止在 RED 阶段跳过测试直接写实现
- **浏览器验证**：单元测试通过后，必须使用 `webapp-testing` skill 在浏览器中实际验证页面功能，确保页面在真实浏览器环境下正常运行

---

## 阶段 07：🔍 代码审查（Code Review）

**触发条件**：每个任务实现完成后，合并前。

**执行要求**：
- 使用 `requesting-code-review` skill 发起代码审查
- 审查维度：代码质量、架构合理性、安全性、性能、可维护性
- **安全审查**：使用 `security-best-practices` skill 对代码进行安全维度审查，网站面临的安全威胁（XSS、CSRF、注入等）远多于其他类型项目，安全审查不可省略
- 严重级别分类：
  - **CRITICAL**：必须修复，阻塞合并（安全漏洞、数据丢失风险、逻辑错误）
  - **IMPORTANT**：强烈建议修复（性能问题、潜在 bug、设计缺陷）
  - **MINOR**：可选修复（代码风格、命名、注释改进）
- CRITICAL 级别问题必须修复后才能继续

---

## 阶段 08：🏁 收尾分支（Finish Branch）

**触发条件**：所有任务完成、代码审查通过后。

**执行要求**：
- **完成前验证**：使用 `verification-before-completion` skill，在声称工作完成之前必须运行验证命令（构建、测试、lint 等）并确认输出通过，禁止未经实际验证就声称完成
- 使用 `finishing-a-development-branch` skill 处理分支生命周期
- 可选操作：
  - **merge**：合并到主分支（适用于已验证的完整功能）
  - **PR**：创建 Pull Request（适用于需要团队审查的场景）
  - **keep**：保留分支继续开发（适用于功能未完全完成）
  - **discard**：丢弃分支（适用于方案废弃或实验失败）
- 合并前确保：所有测试通过、无 CRITICAL/IMPORTANT 级别审查问题、文档已更新
- 清理工作区：删除 `.worktrees/<feature>/` 目录

---

## 工作流执行规则

1. **顺序执行**：阶段 01-08 必须按顺序执行，不可跳过任何阶段
2. **阶段门控**：每个阶段必须有明确的产出物，才能进入下一阶段
3. **回退机制**：若后续阶段发现设计问题，应回退到阶段 01 重新设计
4. **小任务豁免**：对于非常简单的修改（如修复拼写错误、更新配置），可跳过阶段 01-03，直接从阶段 06 开始
5. **紧急修复豁免**：紧急 hotfix 可跳过阶段 01-05，但必须执行阶段 06-08
6. **纯后端豁免**：纯后端/无 UI 项目可跳过阶段 05（前端设计与开发）
